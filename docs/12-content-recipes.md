# 12 · Рецепты добавления контента

Единый чеклист для новых Протоколов, врагов, Модулей/Инъекторов, типов узлов карты и событий. Цель — чтобы контент добавлялся однообразно, а не заново изобретался каждый раз. Раньше это было размазано по `docs/07-архитектура.md`; теперь единственный источник истины — здесь, 07 просто ссылается сюда.

Общее для всех пунктов ниже, не повторяется в каждом разделе:
- Сначала пробуем выразить контент существующими примитивами (`Effect`/`EnemyAction`/`EnemyPattern`). Новый `kind` заводится только когда ни одна комбинация существующих не покрывает задуманное.
- Новый примитив резолвера — сразу юнит-тест в соответствующем `*.test.ts` (не после, не только ручная проверка через `npm run dev`) — см. «Дисциплина проверки» в [07 · Архитектура](07-architecture.md).
- Триггерные/времязависимые механики — юнит-тест проверяет формулу, но живой прогон через Playwright обязателен отдельно (юнит-тесты не ловят баги реальной последовательности ходов в XState-машине — прецедент: Взломанный чип приоритета).
- Обновить соответствующий doc (`03-cards.md`/`04-enemies.md`/`05-items.md`/`06-map.md`) в том же шаге, а не «потом» — иначе доки и код расходятся молча (см. кейс `docs/07`, где комментарий про пул событий не совпадал с кодом до этой ревизии; тот же кейс повторился с `docs/08-roadmap.md` и `docs/11-threat-level.md`, оба чинились отдельной ревизией уже после кода).
- Точный гейт перед тем, как считать шаг законченным — см. [`AGENTS.md`](../AGENTS.md) в корне репо (не дублируется здесь).
- **`id` уникален across всех типов контента, не только внутри своего массива.** Карты, враги, Модули, Инъекторы и узлы карты лежат в разных массивах и резолвятся разными `getXById`, поэтому TS/рантайм не поймает совпадение — обнаружится только руками (или багом в чём-то, что когда-нибудь схлопнёт неймспейсы, например общий React `key` или единый реестр контента). Прецедент: `emp-grenade` был одновременно id карты «РЭБ-граната» и Инъектора «ЭМИ-граната» — оба существовали спокойно, пока их не сверили. Перед тем как завести новый id, `grep -rn "id: \"<new-id>\"" src/data` — быстрее, чем находить это постфактум.

## Новый Протокол (карта)

1. Запись в `src/data/cards.ts` (`CARDS`) — `id`, `name`, `cost`, `type`, `tags`, `description`, `effects: Effect[]`.
2. Эффекты — из уже существующего словаря (`damage`/`block`/`heal`/`gainEnergy`/`applyStatus`/`draw`/`reduceNextCardCost`/`discard`/`doubleNextAttack`), с `Amount` как число или `{ ref, mult? }` для динамических значений. Новый `Effect.kind` — только если ничего из этого не подходит (так появились `discard` и `doubleNextAttack`).
3. Power-карта (`type: "power"`, `trigger`) — не резолвится сразу при розыгрыше, только регистрирует триггер; `triggerAt` — если условие «именно N-я карта за ход» (см. `fireTriggers` в `resolveEffect.ts`).
4. Апгрейд («+»-версия) — отдельная запись с `upgradeOf: <id базовой>`, основной числовой эффект ×1.5 с округлением **в пользу игрока**, Заряд не меняется — кроме именованных исключений из правила апгрейда (docs/03-cards.md: карта теряет Exhaust вместо роста числа, или усиливается статус вместо урона). Не сочинять апгрейд творчески — считать по формуле.
5. Тест в `src/engine/resolveEffect.test.ts`, если эффект новый или комбинация нетривиальна.
6. Обновить `docs/03-cards.md` (таблица архетипа + при необходимости раздел «Правило апгрейда», если это ещё одно исключение).

`REWARD_POOL`/`SHOP` (`runStore.ts`) фильтруют по `!tags.includes("basic") && !upgradeOf` автоматически — новая карта появится в наградах/магазине без правок стора.

## Новый враг

1. Запись в `src/data/enemies.ts` — `id`, `name`, `hpRange: [min, max]`, `moveset: EnemyAction[]`, `pattern`.
2. `EnemyAction` — из существующего словаря (`damage`/`block`/`applyStatus`/`summon`/`damageWithStatus`/`damagePerCardPlayed`). Новый `kind` — тестом в `enemyAi.test.ts` и `resolveEffect.test.ts` (резолвер, а не только intent).
3. `pattern` — `cycle` (детерминированный порядок по `moveIndex`) или `phase` (переключение по HP-порогу, `hpThreshold` в процентах от `maxHp`, см. `enemyAi.ts#resolvePattern`). `weighted` объявлен в типах, но ни один враг им пока не пользуется — если заводишь первого, сверь с `resolvePattern`, что взвешенный выбор там реально детерминирован сидом `state.rng`, а не `Math.random`.
4. Рядовой враг → добавляется в `ENEMIES`; элита/босс → в `ELITE_ENEMIES`/`BOSS_ENEMIES` (`isEliteOrBoss` в этом же файле их оттуда и опознаёт — используется Уровнем угрозы для раздельного HP-множителя, см. [11 · Уровень угрозы](11-threat-level.md)).
5. Если враг встречается в сгенерированной карте — добавить его выбор в `mapGenerator.ts` (`makeNode`, ветка `compartment`/`elite`/`boss`).
6. Обновить `docs/04-enemies.md`.

## Новый Модуль / Инъектор

1. `RelicData` (Модуль, `src/data/modules.ts`) / `PotionData` (Инъектор, `src/data/injectors.ts`) — `id`, `name`, `description`, `effects: Effect[]` (тот же тип, что у карт — резолвится той же `resolveEffects`).
2. Декоративный эффект без реального хука в движке — не считается готовым Модулем. Реальный хук — прямая проверка `state.modules.includes(id)` в конкретной точке резолвера/жизненного цикла хода (образцы в `resolveEffect.ts`: `nanite-reservoir` — `endPlayerTurn`, `reflective-hull` — `onTurnEnd`, `priority-chip` — начало хода, `combat-recorder` — перенос статуса через `runStore.resolveCombat`). Не заводить отдельную подсистему под каждый Модуль.
3. Инъектор — `targeted: boolean` решает, ждёт ли он цель после выбора (см. `SELECT_INJECTOR`/`TARGET_ENEMY` в `combatMachine.ts`).
4. Тест — `resolveEffect.test.ts` для хука Модуля, `runStore.test.ts` для магазинного/наградного появления, если это влияет на пул.
5. Обновить `docs/05-items.md`.

## Новый тип узла карты (комната)

Тяжелее предыдущих пунктов — тип узла зашит в объединение типов и в генератор, а не только в данные. Чек-лист по каждой точке:

1. `src/types/run.ts` — добавить в `MapNodeType`; если узлу нужен свой экран — новое значение `RunScreen`.
2. `src/engine/mapGenerator.ts` — `NODE_LABELS` (подпись), и решить: это случайный узел среднего слоя (`MID_NODE_TYPE_POOL` — вес через повтор в массиве, как у `compartment`), гарантированный хотя бы раз до Стража (как `shop`/`rest` — см. цикл `for (const requiredType of ["shop", "rest"])`), или фиксированная позиция (как `elite`/`boss` — всегда предпоследний/последний слой). `makeNode` — если узел завязан на состав врагов, добавить ветку по образцу `compartment`/`elite`/`boss`.
3. `src/state/runStore.ts` — `enterNode()` должен знать, что делать при входе (сейчас: `screenByType` для простых экранов вроде `rest`/`event`, отдельные ветки для боевых узлов и `shop`). Новый узел с логикой сложнее «просто показать экран» — своя резолвящая экшен-функция в `RunActions`, по образцу `resolveSignalHack`/`buyShopOffer`, плюс `completeNode()` где применимо.
4. Экран для нового узла и его подключение к `switch (screen)` в `App.tsx`, плюс иконка узла (сейчас `NODE_ICONS`/`mapIcons.ts`, см. [10](10-style-pixel.md)) — сейчас это активная зона визуального агента, координировать заранее, не заходить туда параллельно без согласования.
5. `src/engine/mapGenerator.test.ts` — граф остаётся связным (каждый узел следующего слоя достижим, у каждого узла текущего слоя есть исходящее ребро) с новым типом в пуле.
6. Обновить `docs/06-map.md` (граф-схема в начале файла + список типов узлов).

## Новый тип события (Сигнал)

Сейчас реализован только один вариант — «Взломать» (50% кредиты / 50% урон, `resolveSignalHack` в `runStore.ts`). Доки (`docs/01-setting.md`, `docs/06-map.md`) с самого начала описывают пул из нескольких вариантов («найденные данные», «аномалия», «ловушка») — это открытый пункт, а не то, что можно скопировать по образцу один в один: расширение пула требует, чтобы `EventScreen.tsx` умел рендерить разный текст/выбор для разных случайно выбранных вариантов, а этот экран сейчас переписывается визуальным агентом. Строить сторовую логику пула заранее, пока экран рассчитан только на один вариант, смысла нет — контракт данных придётся согласовывать с итоговым UI, а не наоборот. Когда визуальный проход стабилизируется: определить 2–3 варианта как данные (аналогично `ShopOffer`/`EnemyData` — список с `id`/`effects`/текстом), рандомный выбор при входе в `signal`-узел (`enterNode`), рендер в `EventScreen.tsx` по выбранному варианту.
