# 10 · Визуальный стиль B — Пиксельный трофей

> Второй из двух шорт-листнутых кандидатов (см. также [09 · HUD визора](09-style-hud.md)). Это направление уже было заявлено как база в «Тон и визуальный ориентир» [01 · Сеттинга](01-setting.md) — этот документ превращает ту одну строчку в конкретный, реализуемый контракт: токены, компоненты, и отдельно — план на спрайты, потому что в отличие от кандидата A этот стиль **не реализуется одним CSS-проходом**.

## Концепция

Ретро-игра как физический объект трофея: жёсткие пиксельные обводки, никакого сглаживания, блочный HP-бар из сегментов вместо гладкого градиента. Ныряльщик и враги — не текст в рамке, а читаемые силуэты. Это самый «игровой» из кандидатов — ближе к классическому dungeon-crawler/JRPG языку, чем к абстрактному интерфейсу.

Важное отличие от кандидата A: часть этого стиля (цвет, форма панелей, шрифт-имитация) реализуется чистым CSS сразу, но полная версия требует спрайтов (см. «Ассеты» ниже) — Aseprite уже заявлен как инструмент в [08 · Роадмап](08-roadmap.md), фаза 5. Разбить работу на CSS-проход (можно сдать сейчас) и спрайт-проход (зависит от художника/времени на арт) — см. «Порядок реализации».

## Токены

Заменяют `:root` в `src/index.css`, аналогично кандидату A (см. «Как сосуществовать» в конце обоих документов — рекомендуется завести оба набора токенов под `data-visual-style`, не переписывать друг друга).

```css
:root {
  --bg: #131625;          /* индиго-чёрный, нутро корпуса остова */
  --panel: #1d2140;
  --outline: #0a0b14;     /* толстая пиксельная обводка — новый токен, ключевой для этого стиля */
  --ink: #eef1ff;
  --ink-soft: #9195c4;

  --accent: #ffcf5c;      /* Заряд/стоимость/основной интерактив — один акцент, не три как в HUD */
  --danger: #ff5d5d;

  --status-corrosion: #6bff9e;
  --status-overdrive: #ffcf5c;      /* = --accent */
  --status-stabilization: #7fd1ff;
  --status-jamming: #b98bd6;
  --status-breach: #ff5d5d;         /* = --danger */
  --status-reflect: #ff8fd0;

  --corrosion: var(--status-corrosion); /* алиас существующей переменной */

  /* см. "Типографика" — реальный битмап-шрифт либо приближение */
  --sans: var(--mono);
  --mono: ui-monospace, "SF Mono", "Cascadia Code", Consolas, monospace;

  --radius: 0px;          /* пиксель-арт никогда не скруглён */
  --outline-w: 3px;

  color-scheme: dark;
}
```

Цвета статусов намеренно те же **оттенки**, что у кандидата A (зелёный/янтарный/голубой/фиолетовый/красный/розовый) — если решите поддерживать переключение между стилями, распознавание статуса по цвету переносится между ними бесплатно. Здесь они чуть более насыщенные/«игрушечные» — это осознанная разница, не небрежность.

## Типографика

Настоящий битмап-пиксель-шрифт — не встраивать через CDN-ссылку на Google Fonts: это ломается под офлайн PWA (см. `vite-plugin-pwa` в [08 · Роадмап](08-roadmap.md) — service worker кеширует то, что было при первой загрузке, внешний шрифт без сети не подтянется) и вообще внешние сети запрещены в артефактах/строгом CSP, если этот стиль когда-то понадобится демонстрировать так же. Два пути:

1. **Найти шрифт с лицензией, разрешающей самостоятельный хостинг** (например `Press Start 2P`, `04b03`, `Pixel Operator` — все встречаются с открытыми лицензиями), скачать `.woff2`, положить в `src/assets/fonts/`, подключить через `@font-face` с локальным путём (не CDN). Это даёт настоящий битмап-вид у заголовков/чисел.
2. **Если шрифт не нашёлся или не подошёл по лицензии** — приблизить эффект без него: жирный `ui-monospace`, фиксированный маленький размер, и классическая техника «пиксельной обводки текста» через множественный `text-shadow`:

```css
.pixel-outline-text {
  text-shadow:
    1px 0 var(--outline), -1px 0 var(--outline),
    0 1px var(--outline), 0 -1px var(--outline);
}
```

Применить `.pixel-outline-text` на `.enemy-name`, `.card-name`, заголовки `.screen-layout h1`. Числа (`HP`, `Заряд`, цены) — `font-variant-numeric: tabular-nums`, тем же шрифтом.

## Форма панелей — толстая пиксельная обводка

Вместо `border-radius: 6px` (сейчас на `.enemy`, `.card`, `.player-status`, `.reward-card`, `.shop-item`, `.rest-choice`, `.map-node`, кнопки) — жёсткий прямой угол и толстая тёмная обводка, имитирующая пиксельный контур без блюра:

```css
.pixel-panel {
  border-radius: 0;
  border: var(--outline-w) solid var(--outline);
  background: var(--panel);
  box-shadow:
    inset 0 0 0 1px rgba(255, 255, 255, 0.04); /* лёгкий внутренний блик, не градиент */
}
```

Разместить `.pixel-panel` (или продублировать правило) на тех же селекторах, что перечислены для кандидата A — список идентичен, форма другая.

## HP-бар — сегменты, не градиент

```css
.hp-bar {
  height: 8px;
  background: var(--outline);
  border-radius: 0;
}
.hp-fill {
  height: 100%;
  background: repeating-linear-gradient(
    to right,
    var(--status-corrosion) 0 6px,
    color-mix(in srgb, var(--status-corrosion) 80%, black) 6px 8px
  );
  image-rendering: pixelated;
}
```

Даёт классический вид «HP из кубиков» вместо гладкой заливки. Тот же приём подходит для `Заряд`/`Щит`, если в UI появится их визуальная шкала (сейчас это просто текст — оставить текстом, шкала не обязательна).

## Статусы и иконки

Как и в кандидате A, `STATUS_ICONS` (`src/components/statusIcons.ts`) меняет тип с `Record<Status, string>` на `Record<Status, JSX.Element>`. Разница в языке иконки: **не** тонкая линия, а залитый пиксельный силуэт 16×16 с шагом сетки (никакого `stroke-width:2` line-art — здесь заливка блоками).

Если пиксельный шрифт/спрайты ещё не готовы (см. «Ассеты» ниже) — временная замена: залитые геометрические SVG-силуэты (треугольник, ромб, круг) в цвете статуса вместо эмодзи, тот же список, что в кандидате A:
- `corrosion`, `overdrive`, `stabilization`, `jamming`, `breach`, `reflect` — см. чек-лист форм в [09 · HUD визора](09-style-hud.md#статусы-и-иконки), но здесь заливка вместо контура.
- Иконки узлов карты (`NODE_ICONS` в `MapScreen.tsx`) — та же логика, залитые силуэты вместо `⬡◆◈$+☠`.

Это переходное состояние **до** спрайтов, не финал — CSS-проход не должен блокироваться ожиданием арта.

## Ассеты — то, что отличает этот план от кандидата A

Это единственный раздел, которого нет в плане A. Пайплайн: Aseprite → экспорт PNG (индексированная палитра) → `image-rendering: pixelated` при выводе. Каждый спрайт — целочисленное увеличение экрана (1 «игровой» пиксель = N экранных, не дробное масштабирование — иначе размывается).

**Мастер-палитра.** Прежде чем рисовать первый спрайт — зафиксировать общую палитру 16–24 цветов, выведенную из токенов выше (`--bg`, `--panel`, `--outline`, `--ink`, `--accent`, 6× `--status-*`, плюс несколько промежуточных теней/светов на каждый). Все спрайты рисуются в этой палитре — иначе разные существа не будут читаться как одна игра.

**Список спрайтов и приоритет.** Порядок — от того, что игрок видит чаще всего, к тому, что видит реже:

| Приоритет | Существо/объект | Канвас | Примечание |
|---|---|---|---|
| 1 | Ныряльщик (игрок), idle-поза | 32×32 или 48×48 | сейчас не показан визуально вообще — добавление сам факт присутствия портрета уже даст много «сочности» |
| 1 | Санитарный дрон | 32×32 | rank-and-file, см. `src/data/enemies.ts` |
| 1 | Турель обшивки | 32×32 | rank-and-file |
| 1 | Заражённый образец | 32×32 | rank-and-file |
| 2 | Страж-гексапод (элита) | 64×64 | крупнее — читается как более опасный |
| 2 | Ядро-Страж (босс) | 96×96 (или во весь верх экрана) | см. «Фазовый спрайт» ниже |
| 3 | 6× иконки статусов | 16×16 | Коррозия/Форсаж/Стабилизация/Помехи/Пробоина/Отражение |
| 3 | 6× иконки узлов карты | 16×16 или 24×24 | Отсек/Страж/Сигнал/Терминал/Ремонтный отсек/Ядро-Страж |
| 4 (не блокирует v1) | 6× иконки Инъекторов, 4× иконки Модулей | 16×16 | можно оставить текстом на первую итерацию |

**Фазовый спрайт босса — использует уже существующую механику, не новую.** `Ядро-Страж` меняет паттерн атак по HP-порогу (`hpThreshold` в `EnemyPattern`, см. `src/data/enemies.ts` и `docs/04-enemies.md`) — то есть у игры уже есть точный, детерминированный момент «фаза сменилась». Нарисовать **два** варианта спрайта босса (до/после порога) и переключать по тому же условию, что уже переключает паттерн атаки в `enemyAi.ts` (`resolvePattern`) — визуальный и механический «слом» совпадут, это не требует новой логики, только чтение уже существующего состояния (`enemy.hp` относительно `hpThreshold`) на стороне рендера.

**Пока спрайтов нет.** Не блокировать остальную реализацию: показывать плейсхолдер — сплошной силуэт-заглушка (простая геометрическая форма через `clip-path`, в цвете `--panel`/`--outline`, без деталей) на месте будущего спрайта, тем же канвасом/пропорцией, что и финальный арт. Так layout не переверстывается повторно, когда спрайты появятся — они просто встают в готовый слот.

## Компонент за компонентом

Список идентичен таблице кандидата A ([09 · HUD визора](09-style-hud.md#компонент-за-компонентом)) один в один по селекторам — разница только в том, какие правила применяются (`.pixel-panel` вместо `.hud-panel`, толстая обводка вместо среза угла, сегментный `.hp-fill` вместо градиента). Не дублирую таблицу второй раз — открыть план A и построчно заменить трактовку.

Специфичные для B отличия:
- `.card-cost`, `.enemy-name`, `.card-name` — добавить `.pixel-outline-text` (см. «Типографика»).
- `.enemy` — добавить слот под спрайт (`<img>`/`<div class="sprite">` перед именем врага), даже если сейчас туда встаёт только плейсхолдер-силуэт (см. «Пока спрайтов нет»).
- `.player-status` — этому стилю имеет смысл вынести портрет Ныряльщика отдельным блоком над/рядом со статус-баром (сейчас там только текст) — структурное изменение `CombatScreen.tsx`, не только CSS.
- `.injector` (сейчас `border-style: dashed`) — пунктир в пиксель-арте не читается как «дефицитный ресурс» так же чисто, как в HUD; заменить на обычную `.pixel-panel` обводку + маленький бейдж-иконку «расходник» в углу, если время позволяет, иначе оставить как есть.
- `DevPanel.css`/`DevPanel.tsx` — так же, как и в кандидате A: **вне скоупа**, нейтральный служебный вид, не реcкинить.

## Движение

- Смена HP — не плавный transform, а **дискретный** шаг (снять один «кубик» бара за раз, ступенчато, `transition: none` на `.hp-fill`, только мгновенная перерисовка `width`) — это часть ретро-языка, плавность здесь противоречит стилю.
- Появление карты в руке — лёгкий вертикальный «прыжок»-подскок (Framer Motion `keyframes` с overshoot) вместо гладкого fade, ближе к JRPG-меню.
- Урон по врагу — классический «flash» (кратковременная инверсия/побеление спрайта на 1 кадр через CSS `filter: brightness()` анимацию 80–120мс) — дешёвый, узнаваемый приём, не требует Framer Motion.
- Всё выше — обязательно под `@media (prefers-reduced-motion: no-preference)`, статичный fallback без tremors/flash при `reduce`.

## Доступность

- Контраст `--ink` (#eef1ff) на `--bg` (#131625) — высокий, задел есть; `--ink-soft` (#9195c4) на `--panel` (#1d2140) — проверить AA перед сдачей, пиксельные палитры любят быть слишком приглушёнными.
- Толстая обводка (`--outline-w: 3px`) уже сама по себе неплохая замена части `:focus-visible` контраста, но фокус-кольцо всё равно обязательно (жёлтая/акцентная рамка поверх обводки при табуляции).
- `min-height: 44px` — не понижать, как и в кандидате A.

## Порядок реализации

1. **Инфраструктура переключения стиля** — та же, что в п.1 плана A (`data-visual-style="pixel"`), если оба кандидата живут в одной кодовой базе.
2. **CSS-проход (не ждёт художника):** токены, `.pixel-panel`, сегментный `.hp-fill`, `.pixel-outline-text`, силуэты-плейсхолдеры статус-иконок и иконок узлов карты (залитые геометрические формы, см. «Статусы и иконки»).
3. Точечный проход по таблице «Компонент за компонентом» (включая специфичные для B отличия).
4. **Мастер-палитра** — зафиксировать 16–24 цвета из токенов, до начала рисования первого спрайта.
5. **Спрайт-проход по приоритету** (таблица «Список спрайтов») — приоритет 1 (игрок + 3 рядовых врага) раньше приоритета 2/3. Каждый спрайт встаёт в уже готовый слот-плейсхолдер, layout не меняется повторно.
6. Фазовый спрайт босса, читающий тот же `hpThreshold`, что и `enemyAi.ts`.
7. Движение (discrete HP-степ, card-хлоп, damage-flash).
8. Ручная проверка на телефоне/узком вьюпорте, как и в плане A.

## Готово, когда

- [ ] CSS-проход (шаг 2–3) сдан и играбелен **до** того, как готов хоть один спрайт — это отдельная, самостоятельно ценная веха.
- [ ] Мастер-палитра зафиксирована одним файлом/референсом до начала спрайтов.
- [ ] Приоритет-1 спрайты (игрок + 3 рядовых врага) на месте, элита/босс — либо готовы, либо на явном плейсхолдере того же размера.
- [ ] Фаза босса визуально переключается синхронно с реальной сменой паттерна (не рассинхронизирована с `hpThreshold`).
- [ ] Ни одного эмодзи не осталось (та же проверка, что у кандидата A).
- [ ] `npm run dev` на портретном вьюпорте — нет горизонтального скролла, включая самый широкий спрайт (босс 96×96).

## Как сосуществовать с кандидатом A

См. зеркальный раздел в [09 · HUD визора](09-style-hud.md#как-сосуществовать-с-кандидатом-b) — то же самое: токены под `:root[data-visual-style="pixel"]`, общая структурная разметка не дублируется, различается только оформление через токены и несколько сигнатурных правил.
