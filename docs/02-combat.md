# 02 · Боевая петля и мобильный UI

## Боевая петля

Пошаговый бой, один Ныряльщик на игрока. Главный источник тактики — открытый **intent** врага: перед его ходом видно иконку действия (атака + урон, защита, статус), решения по Протоколам принимаются с учётом этого.

1. Заряд (3)
2. Добор руки
3. Розыгрыш Протоколов
4. Сброс руки
5. Ход врага (по intent)

Цикл повторяется до победы или поражения в бою. Вне боя Заряд и Щит обнуляются, HP и статусы (кроме временных) переносятся между отсеками остова.

## Порядок разрешения — фундамент комбинаторики

Раз глубина строится на пересечении эффектов, порядок их срабатывания зафиксирован явно, иначе комбо неоднозначны:

1. **Начало хода игрока:** Щит с прошлого хода спадает (если нет карты с Retain) → срабатывают триггеры `onTurnStart` (Power-протоколы) → добор руки.
2. **Розыгрыш карты:** урон/блок считаются с учётом статусов в момент розыгрыша (Форсаж и Пробоина применяются здесь и сейчас, не задним числом).
3. **Конец хода игрока → начало хода врага:** тик Коррозии для игрока **и всех живых врагов одновременно** (урон по стеку, затем стек −1 — в этом порядке, не наоборот) → уменьшение временных статусов на 1 → срабатывают триггеры `onTurnEnd` → сброс руки.

Тик «разом за раунд», а не индивидуально «в начале хода каждого носителя» — сознательное упрощение реализации: при одном враге разница не ощущается, а с несколькими один общий момент тика проще держать без ошибок очерёдности, чем синхронизировать его с ходом каждого врага по отдельности.

Статус всегда показывается тремя каналами сразу — **иконка + число + цвет**, не только цветом: это требование доступности (дальтонизм) и то, что игрок не обязан запоминать «зелёное = коррозия».

| Статус | Эффект |
|---|---|
| Коррозия | Раз за раунд (см. выше) — урон, равный стеку; стек уменьшается на 1 |
| Форсаж | +X к урону от атакующих Протоколов на оставшийся бой |
| Стабилизация | +X к получаемому Щиту на оставшийся бой |
| Помехи | −25% к наносимому урону, длительность в ходах |
| Пробоина | +50% к получаемому урону, длительность в ходах |

## Мобильный UI и ввод

Портрет — основная ориентация и на телефоне, и на десктопе: окно просто уже, раскладка не перестраивается в ландшафт. Ввод — тап-выбор → тап-цель: коснулся Протокола — он приподнялся и подсветил валидные цели, коснулся цели — сыгран; повторный тап по карте или тап мимо — отмена. Один обработчик работает и для мыши, и для пальца.

Раскладка экрана боя сверху вниз:

```
┌─────────────────────┐
│   ход 3 · пауза      │  верхняя панель
├─────────────────────┤
│                      │
│  враги · intent      │  зона врагов
│                      │
├─────────────────────┤
│  HP · щит · заряд    │  статус игрока
├─────────────────────┤
│  рука протоколов ·   │  рука, скролл,
│  скролл              │  зона большого пальца
└─────────────────────┘
```

- Touch-таргеты не меньше ~44×44px, с отступами между картами в веере, чтобы не задевать соседние при тапе.
- Рука Протоколов — горизонтальный скролл/веер внизу экрана, в зоне досягаемости большого пальца.
- Отступы от системных зон (чёлка, домашняя полоса) — через CSS `env(safe-area-inset-*)`.
- Драг-н-дроп не проектируем как основной путь — при желании можно добавить позже как опциональную надстройку для десктопа поверх тап-модели, не меняя логику эффектов.
- Звук на телефоне инициализируется только по первому касанию пользователя (браузеры блокируют автозвук) — `AudioContext` «будим» на первом тапе, иначе SFX молчат.
