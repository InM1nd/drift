# 07 · Архитектура (Vite + React + PWA)

## Стек

- **Vite + React + TypeScript** — чистый SPA, без Next.js: игре не нужен SSR, только клиентский рендер и офлайн-работа.
- **Состояние боя:** XState — фазы хода как явный автомат (`playerTurn → endingTurn → enemyTurn → victory/defeat`), удобно отлаживать и невозможно попасть в невалидное состояние. Реализовано в `src/engine/combatMachine.ts`.
- **Состояние захода:** Zustand (или React Context) — колода, Модули, Инъекторы, HP, кредиты поверх машины боя.
- **Рендер:** DOM/CSS — пиксель-арт через `image-rendering: pixelated`, Framer Motion для твинов карт, hover и layout-переходов. Canvas/PixiJS — только если позже реально понадобится тяжёлая частичная графика.
- **PWA:** `vite-plugin-pwa` — манифест (иконки, `display: standalone`), service worker кеширует шелл приложения для офлайн-игры, промпт «Установить приложение».

## Модель данных карты

Согласована со списком карт в [03 · Карты](03-cards.md) — 7 атомарных эффектов, включая динамические значения и триггеры Power, а не только фиксированные числа.

```ts
interface CardData {
  id: string;
  name: string;
  cost: number | "X";              // "X" = весь доступный Заряд
  type: "attack" | "skill" | "power";
  tags: string[];
  description: string;             // человекочитаемый текст правил, показывается на карте
  exhaust?: boolean;
  retain?: boolean;
  trigger?: "onTurnStart" | "onTurnEnd" | "onCardPlayed" | "onAttacked";
  triggerAt?: number;               // условие "именно N-я карта за ход" для trigger: "onCardPlayed"
  effects: Effect[];
  upgradeOf?: string;               // id базового Протокола, если это его «+»-версия
}

type Target = "enemy" | "self" | "allEnemies";

type Amount =
  | number
  | { ref: "shield" | "corrosionOnTarget" | "cardsPlayedThisTurn" | "energySpent"; mult?: number };

type Effect =
  | { kind: "damage"; amount: Amount; target: Target }
  | { kind: "block"; amount: Amount }
  | { kind: "heal"; amount: Amount }
  | { kind: "gainEnergy"; amount: Amount }
  | { kind: "applyStatus"; status: Status; stacks: Amount; target: Target; onlyIfPresent?: boolean }
  | { kind: "draw"; count: Amount }
  | { kind: "reduceNextCardCost"; amount: number }
  | { kind: "discard"; count: Amount }
  | { kind: "doubleNextAttack" };
```

> Обновлено по факту реализации Milestone A/B — этот снимок типов теперь соответствует `src/types/card.ts`, а не только первоначальному плану Фазы 1.

`onlyIfPresent` — добавлено при реализации: нужно для карт вида «Катализатор распада» (усилить Коррозию только там, где она уже есть), иначе `target: "allEnemies"` бьёт по всем без разбора и ломает задуманный снежный ком.

Три показательных примера — простая карта, динамическое значение, триггер Power:

```ts
const strike: CardData = {
  id: "strike", name: "Залп", cost: 1, type: "attack", tags: ["basic"],
  effects: [{ kind: "damage", amount: 6, target: "enemy" }],
};

const decayFinisher: CardData = {
  id: "decay-finisher", name: "Финал распада", cost: 2, type: "attack",
  tags: ["corrosion"], exhaust: true,
  effects: [{ kind: "damage", amount: { ref: "corrosionOnTarget", mult: 2 }, target: "enemy" }],
};

const decayCatalyst: CardData = {
  id: "decay-catalyst", name: "Катализатор распада", cost: 2, type: "power",
  tags: ["corrosion"], trigger: "onTurnStart",
  effects: [{ kind: "applyStatus", status: "corrosion", stacks: 1, target: "allEnemies" }],
};
```

Простые карты — чистые данные без специальной логики резолвера; карты со scaling-значением или триггером используют `Amount`-объект/`trigger`, но остаются декларативными записями, а не кодом. Резолвер (как именно `ref` превращается в число, как обрабатывается очередь триггеров) — отдельная задача реализации, не предмет этого документа.

Враг описывается параллельной декларативной структурой — поля `hp` / `moveset` / `pattern`, см. [04 · Враги](04-enemies.md).

## Детерминизм и сохранение посреди боя — обязательное требование мобилки

Это не полировка, а table stakes: телефонную сессию постоянно прерывают в середине боя, и «продолжить точно с того же места» — базовое ожидание. Точное восстановление требует, чтобы вся случайность была детерминированной и сериализуемой вместе с остальным состоянием.

- **Сидированный PRNG вместо `Math.random()`:** вся случайность (замес колоды, дропы, ходы врага) идёт через один генератор с сохраняемой позицией курсора. Это же даёт воспроизводимость сида для реиграбельности.
- **Всё состояние боя — сериализуемый снапшот:** рука, колода, сброс, изгнание, статусы, intent врага и позиция RNG сохраняются в IndexedDB как один объект, из которого бой восстанавливается байт-в-байт.
- **Ловушка XState (замыкания):** в context машины не должно быть замыканий/функций — только сериализуемые данные, иначе снапшот нельзя сохранить и регидрировать. Логику держим в actions/guards, а не в самом состоянии.
- **Ловушка XState (мутация на месте):** найдено на практике при реализации Фазы 1 — если `assign` мутирует вложенный объект контекста и возвращает ту же ссылку, любой ранее сохранённый снапшот (например, в тесте или в компоненте) незаметно меняется вместе с текущим состоянием. Каждый action в `combatMachine.ts` делает `structuredClone(context.combat)` в начале и мутирует уже клон — так снапшоты остаются честными «замороженными» на момент чтения.

## Сохранения

IndexedDB (например, через `idb-keyval`) — прогресс захода хранится локально, бэкенд не нужен.

## Dev-инструменты

Отдельная отладочная панель (только в dev-сборке) — стартовать любой бой, задать HP, выдать Протоколы/Модули/Инъекторы, зафиксировать сид. Без неё сбалансировать декбилдер вручную практически невозможно — строим её рано, а не в конце. Реализовано: `src/components/DevPanel.tsx`. **Не реcкинить** под визуальный стиль (09/10) — это служебный инструмент, не часть игрового опыта.

## Разделение стора: XState vs Zustand

Проверено на практике за Milestone A/B — граница между ними должна оставаться жёсткой, иначе состояние боя и состояние захода начинают путаться:

- **`combatMachine.ts` (XState) владеет всем, что существует только внутри одного боя** — рука, колоды, статусы, HP в течение боя, `activePowerIds`, `injectors` (расходуются из инвентаря забега, но во время боя живут как копия внутри `CombatState`). Контекст — чистые сериализуемые данные (см. «Детерминизм» выше), никакой бизнес-логики о забеге в целом.
- **`runStore.ts` (Zustand + persist) владеет всем, что переживает бой** — колода между боями, кредиты, HP между боями, `ownedModuleIds`, `injectorIds`, карта забега (`mapNodes`/`displayLayers`, генерируется один раз в `createInitialRunState` через `generateMap`), текущий узел, экран (`RunScreen`).
- **Синхронизация — один явный шов, не два стора, слепо читающих друг друга:** `CombatScreen.tsx` строит `CombatState` из снапшота `runStore` при входе в бой (`createInitialCombatState(..., { modules, injectorIds, carriedOverdrive })`) и зеркалит его обратно через `updateActiveCombat` при каждом изменении (нужно и для ресурса, расходуемого в бою, вроде Инъекторов, и для восстановления после перезагрузки посреди боя). `resolveCombat` — единственная точка, где бой формально заканчивается и его результат (HP, Форсаж на перенос) записывается обратно в `runStore` как факт забега.

## Паттерны: добавление нового контента

Не изобретать заново на каждой карте/враге — три реальных паттерна, устоявшихся за Milestone A/B:

- **Новый Протокол** — запись в `src/data/cards.ts` (`CARDS`), эффекты собираются из уже существующих `Effect`-вариантов, если это возможно. Новый `Effect.kind` заводится только тогда, когда ни одна комбинация существующих не покрывает карту (так появились `discard` и `doubleNextAttack`) — и сразу тестом в `resolveEffect.test.ts`, не только вручную через `npm run dev`. Апгрейд («+»-версия): основной числовой эффект ×1.5 с округлением в пользу игрока, `upgradeOf` указывает на базовый id — см. [03 · Карты](03-cards.md).
- **Новый враг** — запись в `src/data/enemies.ts`, `EnemyAction` берётся из существующих вариантов (`damage`/`block`/`applyStatus`/`damageWithStatus`/`damagePerCardPlayed`), `pattern` — `cycle` (детерминированный порядок ходов) или `phase` (переключение по HP-порогу, см. `enemyAi.ts#resolvePattern`). Новый `EnemyAction.kind` — так же, тестом в `enemyAi.test.ts`/`resolveEffect.test.ts`, а не только логами боя.
- **Новый Модуль/Инъектор** — `RelicData`/`PotionData` (`src/types/item.ts`) переиспользуют `Effect[]` через `resolveEffects` (общая точка входа, которую использует и `resolveCardEffects`). Модуль с реальным хуком в движке (не декоративным эффектом) — прямая проверка `state.modules.includes(id)` в соответствующей точке резолвера (см. `nanite-reservoir`/`reflective-hull`/`priority-chip`/`combat-recorder` в `resolveEffect.ts` как образец), а не отдельная подсистема.

## Дисциплина проверки

Правило, которое ловило реальные баги в Milestone A/B (не гипотетическое):

- Каждый новый примитив резолвера/движка — юнит-тест в тот же PR/шаг, не после. Один прогон формулы урона в один ход не ловит баги жизненного цикла статуса за несколько ходов (так был найден и исправлен баг, что Форсаж/Стабилизация тикали вниз наравне с Помехами/Пробоиной, хотя по докам живут «весь бой») — тест должен явно проверять поведение **на границе хода**, если механика вообще связана с ходами.
- Триггерные/времязависимые механики (Power-карты, `onCardPlayed`+`triggerAt`, фаза босса, Инъектор с прицеливанием) — обязательно живой прогон через Playwright (`npm run dev`), не только юнит-тест. Юнит-тест проверяет формулу, Playwright — что реальная последовательность ходов в XState-машине действительно доходит до этой формулы (так был найден баг с Взломанным чипом приоритета — юнит-тесты были зелёными, а скидка не применялась на 1-м ходу боя).
- Перед тем как считать шаг законченным: `npx tsc --noEmit -p tsconfig.app.json`, `npm test`, `npx oxlint src` — все три чистые, не выборочно.
- Коммит — только когда явно попросили. Рабочее дерево может накапливать несколько логических шагов до коммита, если это один согласованный кусок работы (см. историю коммитов Milestone B — B1–B3 одним коммитом, т.к. это была одна одобренная единица плана).

## CSS и визуальные токены

С появлением конкурирующих визуальных направлений ([09 · HUD визора](09-style-hud.md), [10 · Пиксельный трофей](10-style-pixel.md)) это стало жёстким правилом, а не пожеланием: **никаких хардкод-цветов в компонентных `.css`** — только `var(--...)` из `src/index.css`. Любой новый компонент/экран заводит стили через уже существующие токены (`--bg`, `--ink`, `--ink-soft`, `--accent`, `--corrosion`, статусные `--status-*`, если выбран один из двух стилей) — иначе переключение стиля (`data-visual-style`) молча ломается в тех местах, которые забыли завести через переменную.
